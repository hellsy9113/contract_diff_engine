# Contract Diff v2 Word-Diff Audit

This audit covers the current v2 backend flow for:

- `updated draft.pdf` as the original PDF
- `updated draft-1.pdf` as the revised PDF
- `annotated-contract-4.pdf` as the generated annotated output

No behavior was changed as part of this audit.

## 1. Current Pipeline Flow

The production `/compare` endpoint now defaults to v2 unless
`CONTRACT_DIFF_COMPARE_ENGINE=v1` is set.

Current v2 flow:

1. `src/contract_diff_api/routes/compare.py`
   - `compare_documents()` reads `original_file` and `revised_file`.
   - `_compare_v2_response()` calls `compare_pdf_bytes_v2()`.
   - The response is returned as `application/pdf`.

2. `src/contract_diff/services/compare_v2.py`
   - Profiles both PDFs with `profile_pdf()`.
   - Extracts and processes both PDFs with `extract_and_process_pdf()`.
   - Collects comparison blocks with `get_document_comparison_blocks()`.
   - Aligns blocks with `align_structured_blocks()`.
   - Converts matches into `Change` objects with `build_changes_from_alignment()`.
   - Renders the revised PDF with `render_changes_to_pdf()`.
   - Runs `analyze_rendered_pdf()`.
   - Builds `ComparisonReport`.

3. `src/contract_diff/extraction/structured/structured_pdf_reader.py`
   - Uses PyMuPDF `page.get_text("dict")` for blocks, lines, spans, font data,
     and bboxes.
   - Uses `page.get_text("words")` for word-level coordinates.
   - Stores extracted text as `TextBlock`, `TextLine`, `TextSpan`, and
     `ExtractedWord`.

4. `src/contract_diff/extraction/structured/pipeline.py`
   - Runs reading order, column detection, block classification, and section
     path assignment.
   - Returns comparison blocks by excluding headers, footers, and noise.

5. `src/contract_diff/alignment/structured_alignment.py`
   - Aligns whole `TextBlock` objects globally with `difflib.SequenceMatcher`.
   - Produces `BlockMatch` records: `equal`, `insert`, `delete`, `replace`,
     or `uncertain`.

6. `src/contract_diff/comparison/structured_changes.py`
   - Converts block matches into `Change` objects.
   - For modified blocks, calls token diff to compute revised-side
     `changed_fragments`.

7. `src/contract_diff/rendering/pdf_renderer_v2.py`
   - Opens the chosen base PDF bytes, currently revised by default.
   - Searches revised-side fragments on the PDF page with `page.search_for()`.
   - Adds PDF highlight annotations for added/modified changes.
   - Attempts underline marker rendering for deleted changes.

8. `src/contract_diff_api/routes/compare.py`
   - Returns the generated PDF bytes as the HTTP response.
   - There is no persistent backend storage in the API route. Benchmark scripts
     can write outputs to disk, but `/compare` streams the generated bytes.

## 3. Where Paragraph-Level Diffing Is Happening

The main comparison unit in v2 is currently `TextBlock`, not word or sentence.
PyMuPDF decides what a block is. In legal PDFs, a block can be an entire
paragraph, a wrapped clause, a heading plus body text, or sometimes a larger
layout group.

The block-level decision happens in:

- `src/contract_diff/alignment/structured_alignment.py`
  - `align_structured_blocks()`
  - `_align_replacement_segment()`
  - `_block_similarity()`

`SequenceMatcher` receives two lists:

- `original_normalized = [block.normalized_text, ...]`
- `revised_normalized = [block.normalized_text, ...]`

That means the first diff decision is:

```text
original block sequence
vs
revised block sequence
```

Only after a block pair is classified as `replace` does
`structured_changes.py` call:

```python
get_revised_changed_fragments(original_text, revised_text)
```

So the pipeline has token-level fragments, but only inside an already modified
block.

## 4. Why Whole Paragraphs Are Highlighted

Whole paragraphs can be highlighted for several reasons:

1. Added changes use whole revised block text.

   In `src/contract_diff/rendering/pdf_renderer_v2.py`,
   `_search_fragments_for_change()` does:

   ```python
   if change.change_type == "added" and change.revised_text:
       candidates.append(change.revised_text)
   ```

   If block alignment classifies a revised paragraph as inserted, the renderer
   starts from the whole block text.

2. Modified changes fall back to full revised text when fragments are empty and
   confidence is high.

   `_search_fragments_for_change()` also does:

   ```python
   if not candidates and change.revised_text is not None and change.confidence >= 0.9:
       candidates.append(change.revised_text)
   ```

3. Block splitting differences can convert small edits into block inserts or
   deletes.

   If extraction creates one block in the original but two blocks in the
   revised, `SequenceMatcher` may classify one block as deleted and another as
   inserted rather than one local word edit.

4. Rendering searches phrases on the PDF after comparison.

   The renderer does not know the original word alignment. It receives
   `Change.changed_fragments` or block text and searches the PDF for those
   strings. If the search fragment is long or falls back to a block-level
   string, the highlight is broad.

## 5. Why Deleted Text Is Not Visually Represented Correctly

Deleted changes are currently created from block deletes in
`structured_changes.py`.

For a delete:

- `original_location` exists.
- `revised_location` is `None`.
- `revised_text` is `None`.

But `pdf_renderer_v2.py` renders on the selected base PDF, which is revised by
default. `_render_deleted_change()` starts with:

```python
location = change.revised_location

if location is None or location.bbox is None:
    return
```

For true deleted content there is no revised-side bbox, so the function returns
without drawing anything. That means deleted text can exist in the report but
not appear clearly in the generated PDF.

The current renderer also does not create a strikethrough textual annotation
containing deleted words. It only attempts a small underline marker near a
revised anchor, but the current `Change` model does not carry a revised anchor
for deletes.

## 6. Why Annotation Content Is Too Long or Confusing

PDF annotation content is generated in:

- `src/contract_diff/rendering/pdf_renderer_v2.py`
  - `_annotation_content()`

Current behavior:

```python
text = change.revised_text or change.original_text or ""
text = " ".join(text.split())

if len(text) > 140:
    text = f"{text[:137]}..."
```

This means:

- Added annotations use the full revised block text.
- Modified annotations use the full revised block text, not only the changed
  words.
- Deleted annotations use the full original block text.
- Long content is truncated mid-context, which can hide the actual changed
  words.

There is no current annotation text model that says:

```text
one unchanged word before + changed text + one unchanged word after
```

There is also no representation of deleted words as strikethrough text in the
annotation content.

## 7. Page-Position and Page-Number False Positive Risk

The v2 aligner is better than page-by-page comparison because
`align_structured_blocks()` aligns globally across all comparison blocks.
However, page shifts can still create false positives through these paths:

1. Block IDs include page and block index.

   `_block_id()` returns:

   ```python
   f"page-{block.page_index}-block-{block.block_index}"
   ```

   These IDs are used as metadata and not as match keys, but downstream
   locations still depend on page index and block index.

2. Reading order is page-local before global block collection.

   `pipeline.py` processes pages, then `get_comparison_blocks()` flattens
   blocks in page order. If pagination changes split/merge blocks differently,
   global sequence matching can still see inserts/deletes.

3. Header/footer/page-number detection is heuristic.

   `reading_order.py` excludes `header`, `footer`, and `noise`, but if a page
   number or running header is not classified correctly, it can enter the
   comparison block stream.

4. Block-level matching is sensitive to extraction block boundaries.

   A paragraph moved across a page boundary may be extracted as a different
   block grouping. Even if the text is unchanged, the sequence of block strings
   can differ.

5. There is no move operation implementation yet.

   `BlockMatch.operation` allows `"move"`, but `structured_alignment.py`
   currently emits equal/insert/delete/replace from `SequenceMatcher`; it does
   not detect moved text spans or moved sentences as the same content in a new
   location.

## 8. Proposed Implementation Plan: Document-Level Word Diffing

The next implementation should move the primary diff unit from block-level text
to document-level word tokens while preserving source coordinates.

Recommended plan:

1. Add a traceable word token model.

   Suggested fields:

   - `token_id`
   - `text`
   - `normalized_text`
   - `page_index`
   - `block_index`
   - `line_index`
   - `word_index`
   - `bbox`
   - `section_path`

   The source should be `ExtractedPage.words`, enriched with section/block
   metadata from nearby `TextBlock`s.

2. Build document-level token streams.

   Convert both documents into flat word-token streams:

   ```text
   original_tokens = [word, word, word, ...]
   revised_tokens = [word, word, word, ...]
   ```

   Exclude headers, footers, and noise before tokenization.

3. Align token streams globally.

   Use `SequenceMatcher` or a stronger patience/histogram diff over normalized
   word tokens. The comparison should be document-global, not page-local.

4. Group token opcodes into semantic change ranges.

   Convert low-level word opcodes into:

   - added word range
   - deleted word range
   - modified word range
   - unchanged context range

   Group nearby edits only when they are truly adjacent.

5. Preserve exactly one word of context on each side.

   For each changed range, compute:

   - `prefix_context`: one unchanged word before
   - `changed_tokens`
   - `suffix_context`: one unchanged word after

   The context words should be displayed in annotation text but not highlighted
   or crossed out.

6. Represent deleted text explicitly.

   The revised PDF cannot highlight deleted content in place because it does not
   exist there. Instead, deleted words should appear in annotation content as
   strikethrough text and optionally be attached to the nearest revised anchor
   word or block.

   Suggested annotation text model:

   ```text
   context_before + deleted_text(struck) + context_after
   ```

7. Update `Change` to carry word ranges.

   Add revised/original token references:

   - `original_token_ids`
   - `revised_token_ids`
   - `context_before`
   - `context_after`
   - `annotation_summary`
   - `deleted_text`

8. Render from token bboxes instead of searching full paragraphs.

   For added/modified revised tokens, use actual revised token bboxes.
   This avoids `page.search_for()` matching repeated phrases elsewhere and
   prevents broad paragraph highlights.

9. Add deleted annotation rendering.

   Since deleted text is absent from the revised PDF, render a small, safe
   anchor annotation near the nearest unchanged revised token and put the
   deleted content in the annotation body with strikethrough markup if supported
   by the chosen PDF annotation type. If PDF rich text support is unreliable,
   use plain text with a clear convention first, then add richer rendering.

10. Add page-shift regression tests.

    Test unchanged text moving from page 2 to page 3, text reflowing across page
    boundaries, and repeated page headers/footers. The expected result should be
    no added/modified changes for unchanged body text.

## 9. Immediate Risk Summary

Current v2 is useful for proving the full pipeline, but the comparison unit is
still too coarse for production-grade legal redlines. The most important next
change is not a rendering tweak; it is to make the comparison layer produce
word-level, traceable change ranges before rendering begins.
